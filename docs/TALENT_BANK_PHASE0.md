# Talent Bank — Phase 0 Decisions

> Companion to `taxonomy.seed.json` (455 nodes, generated by `scripts/generate_taxonomy.py`).
> This records the structural decisions applied when converting the planning document
> into seed data, so future edits don't silently reintroduce resolved ambiguities.

## 1. Canonical layer definitions

Each layer answers exactly one question. If a concept seems to fit two layers, these
definitions decide — not intuition at data-entry time.

| Layer | Type key | Answers | Picked by |
|---|---|---|---|
| Industry | `industry` | What commodity/market does the work serve? | Both (job + resume, multi) |
| Business Vertical | `vertical` | What kind of company is the employer? | Company profile (single) |
| Functional Domain | `domain` | Which department pays this person? | Both (single, sub-domain optional) |
| Job Family | `family` | Which career ladder is this? | **Never picked directly — derived from Role** |
| Role | `role` | What is the seniority-neutral job title? | Both (single, "Other" free text allowed) |
| Specialist Discipline | `discipline` | What standards/activities does the person know? | Resume (multi) · Job (multi, as requirements) |
| Equipment Expertise | `equipment` | What hardware has the person worked on? | Resume (multi) · Job (multi, optional) |
| Certification | `certification` | What credential does the person hold? | Resume (multi) · Job (multi, as requirements) |

**Key rule:** the source document listed roles under Domains *and* Families. Resolved:
domains carry **no roles** (department tags + sub-domains only); every role lives under
exactly one family. Family is auto-derived from the chosen role, so users never face
the domain-vs-family ambiguity.

**Name collisions across layers are intentional and safe.** "Operations" exists as
`dom-operations` and `fam-operations`; "Inspection" as a vertical, a domain, and a
family. IDs are type-scoped, and the UI always presents one layer at a time.

## 2. Seniority

Roles are seniority-neutral ("Senior Operator" → role `Operator` + seniority `Senior`).
Seniority is a fixed enum, not taxonomy: Trainee, Junior, Mid-level, Senior, Lead,
Manager, Senior Manager, Head / Director. Rationale: prevents ladder-explosion in the
role list and lets matching score "role exact + seniority adjacent" separately.

## 4. Employment types

Extended beyond the current Jobs enum to match industry hiring reality:
**Permanent, Contract, Turnaround, Shutdown, Part-time, Internship.**
⚠️ This changes the `type` enum in `isValidJob` (firestore.rules) and the Jobs modal
chips — a Phase 2 change requiring a rules deploy. "Full-time" maps to "Permanent";
"Freelance" folds into "Contract".

## 5. Mandatory vs optional (the form-friction policy)

Principle: 3 required picks; everything else optional-but-scored, pitched as
"every field you add improves your match score." Absence scores 0 ("no evidence"),
never negative.

**Blueprint required:** Industry (≥1) · Domain (1) · Role (1, or "Other" free text).
**Blueprint optional:** seniority, sub-domain, certifications, disciplines, equipment,
sector-experience list, years of experience, preferred locations, employment types.

**Job post required:** title, description, company, Industry (≥1), Domain (1), Role (1).
**Job post optional:** seniority sought, required certifications, required disciplines,
equipment context, employment type, location, salary.

## 6. Storage decisions (for Phase 1)

Single `taxonomy` collection, one doc per node, mirroring the proven
`company_categories` shape: `{id, type, name, parentId, level, aliases[], order}`.
Doc IDs = the stable slugs from the seed (`role-terminal-operator`), so references
in resumes/jobs are human-readable and rename-safe (renames touch `name` only, never
the ID). Composite index: `taxonomy(type ASC, order ASC)`. Admin CRUD extends the
existing category-manager pattern.

**`company_categories` is permanently separate from the taxonomy — by design, not
deferral.** They answer different questions: `company_categories` = what products/
services a company *sells* (commercial directory axis); the taxonomy = career and
matching concepts. A company carries both without conflict — e.g. categories
"Tank Seals, Rim Seals" *and* taxonomy vertical "Equipment OEMs". In Phase 2 the
Jobs/Blueprint "Domain Experience" pickers move from `company_categories` to the
taxonomy; the company directory keeps `company_categories` untouched.

**Extensibility rule:** adding nodes/aliases and renaming display names is always
safe (IDs never change). Deleting or merging nodes after resumes/jobs reference
them requires a data migration — add freely, delete carefully.

Rubric weights move to `settings/matching` (admin-editable). "Authorised partners"
editing parameters = a third permission tier that doesn't exist yet — explicitly
deferred, not forgotten.

## 7. Deferred (decided *against* for v1)

Multi-parent nodes (single parent + aliases instead) · semantic/embedding matching
(exact-tag first; revisit only if tag matching proves insufficient) · partner
permission tier · example company names as vertical aliases (Vopak etc. are
examples, not synonyms).
